# Guest-First Auth + Email Magic Link Plan (Decision-Complete, AI-Implementer Ready)
## Supabase Auth, Guest + Magic Link First (Phone/Google Later)
## Future-Compatible with Quick Guest Mode and Ongoing Room Ideas

## Summary

This plan replaces the previous Guest + Google-first auth plan with a faster, lower-friction path: **Guest + Email Magic Link** using Supabase Auth. It removes backend/provider jargon from the UI (no “Supabase login” language), adds a primary **Continue as Guest** path, adds **Email me a sign-in link** as the main non-guest option, preserves the existing meeting engine and ritual flow, and keeps the architecture compatible with future **Phone OTP**, **Google OAuth**, **Quick Guest Meeting**, and **ongoing/perpetual room** work. Under the hood, guest mode uses **Supabase anonymous auth** so the current database constraints and continuity model stay intact without fake IDs or a parallel guest identity hack.

## Revision Notes (Post-Review Tightening)

### Revision Pass A (Protocol + Semantics)
1. Tighten callback-route requirements to support the actual Supabase email-link callback variants (PKCE code exchange preferred, OTP token-hash fallback supported if needed).
2. Tighten `join` bootstrap semantics so it checks `getUserById(...)` first and only bootstraps on `NOT_FOUND`.
3. Clarify anti-enumeration behavior so UI success/error states do not leak account existence but still surface true environment/config breakage.
4. Require sanitized server-side logging for callback/bootstrap failures.

### Revision Pass B (Execution + Rollout Discipline)
1. Add explicit verification evidence expectations (commands + outcomes), not just a test list.
2. Add magic-link cooldown/rate-limit handling expectations for UX and error messaging.
3. Add rollback guidance (disable/hide magic-link CTA while keeping guest/password fallback if rollout is unstable).
4. Tighten acceptance criteria to require one real callback-path success in a configured environment.

## Why This Exists (Product Problem in Plain English)

The current landing/auth experience leaks infrastructure language (“Supabase account”) into a user-facing moment that should feel like entering a room, not logging into tooling. At the same time, the app’s continuity features (meeting history, callbacks, memory) rely on user identity. We need a lower-friction, more human sign-in path that still honors the current data model. Email magic links are simpler to ship than Google OAuth and avoid the Google Console / consent-screen setup overhead while still giving users a real account path beyond guest mode.

## Under-the-Hood Auth Architecture (Final)

## Overview
All auth entry paths must converge on the same outcome:
1. Valid Supabase auth session cookies
2. `locals.userId` resolvable via current hook/auth adapter
3. `public.users` row exists for that auth user id
4. Existing meeting creation and continuity systems proceed unchanged

## Auth Entry Paths

### A. Guest flow (new)
- UI button posts to `?/continueGuest`
- Server action uses Supabase anonymous auth
- Server sets cookies
- Server ensures `public.users` row with `is_anonymous = true`
- Redirect to `/`

### B. Email magic-link flow (new primary non-guest path)
- UI form submits email to `?/sendMagicLink`
- Server action calls Supabase `signInWithOtp` for email
- Sends magic link to user email with redirect back to app callback route
- Server returns success UI message (`Check your email for a sign-in link`)
- User clicks email link
- Supabase redirects back to app (`/auth/callback`)
- Callback finalizes sign-in using the configured Supabase callback shape (PKCE `code` exchange preferred; token-hash verification fallback supported if needed) and sets cookies
- Callback ensures `public.users` row with `is_anonymous = false`
- Redirect `/`

### C. Email/password (existing, fallback)
- Existing `signIn` action remains hidden/secondary
- After successful sign-in and cookie set, call same `ensureUserProfile(...)`
- Redirect `/`

### D. Join meeting (existing, hardened)
- `join` action defensively checks `getUserById(...)` first and only calls `ensureUserProfile(...)` when the profile row is missing
- This protects against first-login/profile-race failures
- Then calls `createMeeting(...)`

## Auth Flow Details (Decision Complete)

## A. Continue as Guest (anonymous auth)

### User flow
1. User clicks `Continue as Guest`
2. Server action `continueGuest` creates anonymous Supabase auth session
3. Server sets auth cookies
4. Server ensures `public.users` row (`display_name = 'Guest'`, `is_anonymous = true`)
5. User returns to `/` in signed-in guest state
6. User can start meeting normally

### Failure behavior
- If anonymous sign-in fails: return friendly error message on landing page
- If cookie set succeeds but profile ensure fails: clear cookies and return blocking error

## B. Email Magic Link (email OTP / link)

### User flow
1. User enters email and submits `Email me a sign-in link`
2. Server action `sendMagicLink` validates email format
3. Action calls Supabase `signInWithOtp({ email, options: { emailRedirectTo: <origin>/auth/callback } })`
4. UI shows success state (no user enumeration details)
5. User clicks email link
6. App callback route finalizes session (`exchangeCodeForSession` or equivalent Supabase callback completion flow)
7. Callback sets auth cookies
8. Callback ensures `public.users` row (`is_anonymous = false`, displayName derived safely)
9. Redirect `/`

### Security/UX rules
- Do not leak whether email is already registered
- Use the same success copy for normal request outcomes whenever practical (anti-enumeration posture)
- Only show a distinct error when the request is invalid (e.g. malformed email) or the environment/config is actually broken
- No raw Supabase error text in UI

### Callback route behavior (exact)
Callback route must:
1. Parse query params (`code`, `error`, optional `error_description`)
2. If `error` present -> redirect `/` with friendly message
3. If `code` present -> call Supabase PKCE session exchange/completion
4. Else if token-hash style params are present (`token_hash` + `type`) -> verify OTP with Supabase and complete session
5. Else -> treat as invalid callback and redirect with friendly message
6. Validate session/user exists
7. Set auth cookies
8. Ensure `public.users` row
9. Redirect `/`
10. On failure after partial session setup -> clear cookies and redirect with friendly message
11. Log a sanitized callback failure reason server-side for debugging (no raw provider text in user UI)

## C. Email/password fallback (existing)

### Behavior in this pass
- Keep the action working
- Hide it behind `More sign-in options`
- After successful sign-in, call `ensureUserProfile(...)`
- Redirect `/`

Rationale:
- Avoid regressions while the new primary flow settles

## Identity/Profile Bootstrap Design (Important)

## Why no shared guest bucket
Rejected explicitly:
- A shared guest account would mix strangers’ continuity/memory/callbacks
- That is unsafe/confusing for this product
- It also makes future cleanup/migration worse

## Approved continuity model
Per-browser/per-session guest continuity using anonymous auth:
- Each guest gets a unique auth user + `public.users` row
- Continuity persists only as long as that auth session/cookies survive
- If cookies are cleared or guest signs out, continuity is lost (acceptable for guest mode)

## `ensureUserProfile(...)` is NOT profile sync
This is the key anti-bug rule:
- It exists to ensure a required row exists for DB integrity
- It is **not** the place to sync provider metadata or overwrite user-customized profile values

## SetupFlow `userName` is meeting-local alias
- The name entered in setup is used for the meeting experience/transcript context
- It does not mutate `public.users.display_name` in this pass
- If we later want “remember this name,” that becomes a separate explicit feature

## Database Bloat / Guest Cleanup (Accepted Debt + Future Work)

## Known tradeoff (accepted for now)
Every guest session can create:
- `auth.users` anonymous user
- `public.users` row
- meetings/shares/etc.

If guests try once and disappear, this creates “dead” guest rows over time.

## Decision for this pass
Accept this debt for now because:
- current user count is effectively zero
- product velocity matters more than preemptive cleanup systems
- `public.users.is_anonymous` makes cleanup straightforward later

## Required follow-up issue (must be tracked)
Create a post-ship issue for guest cleanup:
- purge inactive anonymous users after a retention window (30/60/90 days)
- handle cascades/related records safely
- define whether analytics retention is needed before deletion

1. Merge code changes behind normal PR/CI process.
2. Confirm Supabase project settings for:
   - anonymous auth enabled
   - email auth / magic link enabled
   - redirect URL allowlist includes `/auth/callback`
3. Deploy app.
4. Smoke test (in order):
   - Guest sign-in
   - Email magic-link request
   - Magic-link callback sign-in completion
   - Email/password fallback (still works)
   - Join meeting
   - Share / close / reload
5. Monitor logs for:
   - callback failures
   - profile bootstrap failures
   - auth cookie/session parsing regressions
   - meeting create FK failures
6. If magic-link rollout is unstable, keep guest + password fallback active and temporarily hide/disable the magic-link CTA until callback/config issues are fixed.

No new DB migration is required specifically for this auth UX change because `public.users.is_anonymous` already exists.

## Risks and Mitigations

## Risk 1: Missing `public.users` row causes meeting creation failure
**Mitigation:** Add `ensureUserProfile(...)`; call it in all auth success paths and defensively in `join`.

## Risk 2: Magic-link callback flow differs in environment/config and fails silently
**Mitigation:** Add explicit callback route tests, friendly error handling, and deployment smoke test for real callback path.

## Risk 3: Anonymous auth enabled locally but not in production
**Mitigation:** Pre-deploy checklist requires Supabase auth provider settings verification.

## Risk 4: Guest account bloat accumulates over time
**Mitigation:** Accept now; track and implement anonymous-user cleanup job later.

## Risk 5: AI implementer “fixes copy only” and claims feature done
**Mitigation:** Acceptance criteria require working guest + magic-link flows, callback route, profile bootstrap, and tests.

## “Don’t Be Bullshittin” Reminder for the AI Implementer

## Do NOT take these shortcuts
1. Do not stop at copy changes; this feature is not “done” until guest + magic-link + callback + profile bootstrap work.
2. Do not invent fake guest IDs in forms/cookies to bypass auth; use real anonymous auth sessions.
3. Do not create a shared guest account/bucket for all guests.
4. Do not assume `public.users` rows auto-appear from Supabase auth; the repo does not show a trigger for this.
5. Do not bypass the DB seam by sprinkling direct profile writes across route handlers.
6. Do not overwrite user-editable profile fields inside `ensureUserProfile(...)` (bootstrap-only means bootstrap-only).
7. Do not treat SetupFlow `userName` as account profile sync in this pass.
8. Do not expose raw Supabase/auth errors to users.
9. Do not break the hidden email/password fallback while adding guest + magic link.
10. Do not claim magic-link sign-in works without implementing and testing the callback route.
11. Do not claim guest continuity works unless you verify session persistence across reload in the same browser.
12. Do not add phone auth opportunistically in this PR “while you’re in there”; phone is deferred on purpose.
13. Do not call `ensureUserProfile(...)` and ignore the `SeamResult`; every bootstrap path must check `.ok`.
14. Do not blindly call `ensureUserProfile(...)` in `join`; check `getUserById(...)` first and only bootstrap on `NOT_FOUND`.

## What “done” actually means here
- Users can enter as guest or request an email sign-in link without seeing backend jargon.
- Auth completion leads to a valid session and a bootstrapped `public.users` row.
- Meetings start normally for guest and signed-in users.
- Tests prove the new auth/bootstrap paths.
- Future phone/Google/quick-guest work is not made harder by today’s implementation.
